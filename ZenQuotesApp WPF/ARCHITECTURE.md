# ZenQuotesApp — описание проекта

Десктоп-приложение «генератор цитат» на **WPF (.NET 8)**. Загружает цитаты из стороннего
API [zenquotes.io](https://zenquotes.io), хранит их в локальной базе **SQLite**, оформлено
в тёмной теме в стиле Spotify. Документ описывает назначение каждого файла, использованные
механики и причины принятых решений.

---

## 1. Архитектура в целом

Приложение построено по шаблону **MVVM (Model–View–ViewModel)** и разбито на слои:

```
View (XAML)         MainWindow.xaml + App.xaml         — интерфейс и стили
   │  data binding / commands
ViewModel           MainViewModel + RelayCommand        — состояние и логика экранов
   │
Services            QuoteService, QuoteApiClient,        — бизнес-логика, сеть, настройки
                    SettingsStore, BackupQuotes
   │
Data                QuoteRepository                      — доступ к SQLite
   │
Model               Quote                                — структура данных
```

**Почему MVVM:** разметка (View) не знает о деталях логики, а логика (ViewModel) — о визуале.
Связь идёт через привязки данных (`Binding`) и команды (`ICommand`). Это делает код тестируемым,
читаемым и соответствует «правильной» архитектуре WPF-приложений.

## 2. Конфигурация

### `ZenQuotesApp.slnx`
Файл решения в новом XML-формате. Содержит ссылку на единственный проект `ZenQuotesApp.Wpf`.

### `ZenQuotesApp.Wpf/ZenQuotesApp.Wpf.csproj`
Настройки проекта:
- `OutputType=WinExe` — оконное приложение (без консоли);
- `TargetFramework=net8.0-windows` — WPF доступен только на Windows-таргете;
- `UseWPF=true` — подключает сборки WPF;
- `Nullable=enable` — компилятор предупреждает о возможных `null` (меньше ошибок);
- `ImplicitUsings=enable` — частые `using` (System, System.Linq и т.д.) подключаются автоматически;
- `PackageReference Microsoft.Data.Sqlite` — лёгкий официальный драйвер SQLite.

## 3. Модель данных

### `Models/Quote.cs`
Простой класс с двумя свойствами — `Text` (текст цитаты) и `Author` (автор).
**Почему так просто:** это «единица данных», которая ходит через все слои (API → БД → UI).
Никакой логики в модели нет — она остаётся чистой структурой.

---

## 4. Слой данных (SQLite)

### `Data/QuoteRepository.cs`
Инкапсулирует всю работу с базой через `Microsoft.Data.Sqlite`.

- **Инициализация** — при создании создаёт файл БД (если нет) и таблицу
  `Quotes(Id, Text UNIQUE, Author)`.
- **`UpsertMany`** — вставляет цитаты командой `INSERT OR IGNORE` внутри **транзакции**.
- **`GetAll` / `Count`** — чтение всех цитат и их количества.

**Механики и почему:**
- **`UNIQUE` по тексту + `INSERT OR IGNORE`** — дедупликация на уровне БД: повторная цитата
  просто игнорируется, не нужно проверять дубликаты в коде.
- **Транзакция** на пакетную вставку — десятки/сотни `INSERT` выполняются на порядок быстрее,
  чем по одному.
- **Параметризованные запросы** (`@text`, `@author`) — защита от SQL-инъекций и корректная
  обработка кавычек в тексте.
- Файл БД лежит в `%LOCALAPPDATA%\ZenQuotesApp\quotes.db` — это «правильное» место для данных
  пользователя (не рядом с exe, не требует прав администратора).

## 5. Слой сервисов

### `Services/QuoteApiClient.cs`
Клиент HTTP к `https://zenquotes.io/api/quotes`.

- **Статический `HttpClient`** с таймаутом 60 c и заголовком `User-Agent` (браузерным).
- `FetchAsync` делает GET-запрос, проверяет, что ответ — это JSON-массив, десериализует его
  (`System.Text.Json`) и превращает поля API `q`/`a` в `Quote`.
- Отсеивает служебную «цитату»-заглушку от API при превышении лимита (автор `zenquotes.io`).

**Почему так:**
- Один **переиспользуемый** `HttpClient` (а не новый на каждый запрос) — иначе исчерпываются
  сетевые сокеты.
- **Таймаут** не даёт интерфейсу «висеть», если сеть медленная/недоступна.
- `User-Agent` — без него сайт иногда отвечает блокировкой/капчей.

### `Services/BackupQuotes.cs`
Возвращает 8 «зашитых» цитат на крайний случай. **Почему:** если API недоступен и база пуста,
приложение всё равно должно что-то показать, а не пустой экран. В базу резерв не пишется.

### `Services/QuoteService.cs`
Оркестратор загрузки — главная бизнес-логика.

- **`LoadAsync(batchCount, apiKey, progress, ct)`** — делает `batchCount` запросов **подряд**
  (мультибатч), накапливая уникальные цитаты в `Dictionary` (дедуп по тексту).
- Между запросами без ключа выдерживает паузу ~6 c (лимит ZenQuotes ~5 запросов/30 c).
- Сообщает прогресс через `IProgress<string>` («запрос 2 из 5…»).
- **Стратегия фолбэка:** успех API → сохранить в БД и вернуть всё из БД; иначе → если в БД
  что-то есть, вернуть из БД; иначе → резервные цитаты.
- Запись в БД делается **без токена отмены** — даже при отмене уже скачанное сохраняется.
- `QuoteSource` (enum) и `LoadResult` — сообщают ViewModel, откуда пришли данные и сколько добавилось.

**Почему так:** это и есть «продуманная логика загрузки» — приложение всегда работоспособно
(сеть → кэш → резерв), не зависает (всё асинхронно), не дублирует данные и не упирается в лимит API.

### `Services/AppSettings.cs`
Класс настроек: `BatchCount` (сколько запросов за обновление) и `ApiKey` (необязательный ключ).

### `Services/SettingsStore.cs`
Загрузка/сохранение `AppSettings` в `settings.json` (`System.Text.Json`) рядом с базой.
При битом файле возвращает значения по умолчанию. **Почему JSON-файл:** настроек мало,
заводить под них таблицу избыточно; человекочитаемый файл проще.

## 6. Слой ViewModel (MVVM)

### `ViewModels/ViewModelBase.cs`
Базовый класс с реализацией `INotifyPropertyChanged` и помощником `SetProperty`.
**Почему:** интерфейс автоматически обновляется, когда меняется свойство — это «двигатель»
привязок данных в WPF. `SetProperty` убирает рутину (сравнить, присвоить, уведомить).

### `ViewModels/RelayCommand.cs`
Реализация `ICommand` — позволяет привязывать действия кнопок к методам ViewModel
(`Command="{Binding ...}"`), с поддержкой `CanExecute` (доступность кнопки).
Подписка на `CommandManager.RequerySuggested` автоматически пересчитывает доступность команд.

### `ViewModels/MainViewModel.cs`
Главная вью-модель — состояние и логика всех экранов.

- **Навигация:** `CurrentSection` (enum Home/Search/Game/Library/Settings) + булевы `IsHome`…,
  к которым привязана видимость панелей.
- **Коллекции** `ObservableCollection`: `Library`, `SearchResults`, `GameOptions` — при изменении
  список в UI обновляется сам.
- **Команды:** Navigate, NextRandom, Search, Refresh, Answer, NewGame, SaveSettings, Cancel.
- **Загрузка** `LoadAsync` — создаёт `CancellationTokenSource`, зовёт сервис, через `Progress`
  пишет статус, заполняет данные в `ApplyResult`.
- **Отмена:** `CancelCommand` дёргает `CancellationToken`; кнопка видна только во время загрузки.
- **Поиск** — по тексту и автору, по уже загруженной коллекции, регистронезависимо
  (`StringComparison.OrdinalIgnoreCase`).
- **Игра «угадай автора»** — берёт случайную цитату, подмешивает 3 чужих автора, перемешивает.
- **Настройки** — читаются/пишутся через `SettingsStore`.

**Почему так:** вся логика экранов в одном месте, без кода в разметке. Асинхронность +
`CancellationToken` + `IProgress` дают отзывчивый интерфейс с живым прогрессом и возможностью отмены.

## 7. Слой View (интерфейс)

### `App.xaml` / `App.xaml.cs`
Точка входа приложения и **глобальный словарь ресурсов**: палитра Spotify (тёмные тона + зелёный
акцент `#1DB954`), текстовые стили, стили кнопок (навигация, акцентная «пилюля», вторичная,
вариант ответа, кнопки заголовка), поле поиска, **тонкая полоса прокрутки** и **стиль списка с
виртуализацией**. Здесь же конвертер `BooleanToVisibilityConverter`.
**Почему ресурсы в одном месте:** единый стиль для всего приложения, без дублирования в разметке.

### `MainWindow.xaml`
Главное окно. Ключевые части:
- **`WindowChrome`** — кастомная тёмная полоса заголовка со своими кнопками свернуть/развернуть/
  закрыть (вместо системной светлой), при сохранении перетаскивания и изменения размеров.
- **Сайдбар** слева с навигацией (Главная/Поиск/Игра/Библиотека/Настройки).
- **Контент** — пять панелей, переключаемых через привязку `Visibility` к `IsHome`/`IsSearch`/…
- **Списки `Библиотека` и `Поиск`** — это `ListBox` со включённой **виртуализацией**
  (`VirtualizingPanel`), поэтому рендерятся только видимые строки — список выдерживает десятки
  тысяч элементов без подвисаний.
- **Статус-бар** снизу: источник данных, число цитат, индикатор «Загрузка…» и кнопка «Отмена».

### `MainWindow.xaml.cs`
Code-behind окна:
- **Композиционный корень** — создаёт `QuoteRepository`, `QuoteApiClient`, `QuoteService`,
  `SettingsStore`, собирает `MainViewModel` и ставит его в `DataContext`.
- При `Loaded` запускает первичную загрузку (`InitializeAsync`).
- Обработчики кнопок окна (свернуть/развернуть/закрыть) и `OnStateChanged` (компенсация отступа
  в развёрнутом окне + смена глифа развернуть/восстановить).
- `Hyperlink_RequestNavigate` — открывает ссылку (на страницу ключей API) в браузере.

**Почему сборка зависимостей в коде окна:** приложение небольшое, полноценный DI-контейнер
избыточен; ручная сборка проще и наглядна.

## 8. Сквозные механики (и зачем)

| Механика | Где | Зачем |
|---|---|---|
| MVVM + Binding + `ICommand` | весь UI | разделение разметки и логики |
| `async/await` + `Task.Run` | загрузка, работа с БД | интерфейс не зависает |
| `CancellationToken` | загрузка, кнопка «Отмена» | прервать долгую загрузку |
| `IProgress<string>` | мультибатч | живой прогресс в статус-баре |
| `ObservableCollection` + `INotifyPropertyChanged` | списки и поля | авто-обновление UI |
| Виртуализация (`VirtualizingPanel`) | Библиотека, Поиск | большие наборы без тормозов |
| SQLite + `UNIQUE`/`INSERT OR IGNORE` | хранилище | хранение и дедуп цитат |
| `System.Text.Json` | API и настройки | разбор/сохранение данных |
| `WindowChrome` + `ControlTemplate`/`Style` | окно, тема | оформление в стиле Spotify |

---

## 9. Поток данных

**При запуске:** окно создаёт сервисы → `MainViewModel.InitializeAsync` → один запрос к API →
успех: сохранить в SQLite и показать всё из БД; неудача: показать из БД; пусто: резервные цитаты.

**По кнопке «Обновить»:** `batchCount` запросов подряд (с паузами без ключа), прогресс в статус-баре,
новые уникальные цитаты добавляются в БД, библиотека обновляется. Загрузку можно отменить —
уже скачанное сохранится.
